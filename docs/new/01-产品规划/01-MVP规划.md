# MVP 规划

> 版本 v1.0

## 第一部分:MVP 目标

MVP 的目标不是实现所有设想,而是在最短的时间内做出一套真正可用的数字员工系统,这套系统必须能够创建数字员工,能够让数字员工持续工作,能够让多个数字员工协同完成任务,一句话就是先做出你自己每天都愿意打开使用的版本。

## 第二部分:MVP 产品目标

发布第一个可运行版本。支持创建 Agent,管理员工,Agent 聊天,记忆系统,知识库,工具调用,工作流,桌面端和 Web 管理后台,让创业者可以每天依赖它完成一部分真实工作。

## 第三部分:MVP 用户

第一批用户是独立开发者、一人公司、AI 创业者、软件外包团队、小型工作室。

MVP 阶段不追求覆盖所有行业,而是优先解决一个核心问题:一个人如何拥有多个数字员工。

## 第四部分:功能范围 P0

必须完成的功能包括:

- **用户系统**:注册、登录、用户中心、组织空间
- **Agent 管理**:创建、删除、编辑、查看和状态管理
- **Agent Chat**:多轮对话、文件上传、工具调用状态展示
- **知识库**:支持 PDF、Word、Markdown、TXT,支持上传、删除、更新、检索
- **记忆系统**:短期记忆和长期记忆,用户画像、项目记录
- **工具调用**:GitHub、浏览器、HTTP API、数据库、MCP
- **工作流**:单 Agent 工作流、多 Agent 协作、人工确认节点
- **Dashboard**:今日任务、Agent 状态、任务统计
- **桌面端**(优先级高):跨平台桌面壳、通知和基础桌面助手
- **Web 管理后台**:用户管理、Agent 管理、日志

## 第五部分:P1

第二阶段包括:Agent Forge、3D Agent、技能芯片、成长系统、蓝图、Marketplace、桌面宠物、移动端。

## 第六部分:P2

第三阶段包括:开放平台、插件市场、企业版、开放 API。

## 第七部分:明确暂不进入 MVP 的功能

包括:Marketplace、复杂 3D 动画、实时数字办公室、开放给第三方的开发者平台。

## 第八部分:MVP 验收标准

- 用户可以创建多个 Agent
- Agent 可以访问知识库并完成问答
- 可以调用至少一种外部工具
- 两个以上 Agent 可以完成一次协作流程
- 新用户无需阅读文档即可上手使用
- 创业者可以至少跑通一个典型业务流程,例如客户咨询到报价到合同

## 第九部分:建议开发顺序

- **Sprint 1**:用户系统、基础聊天、Agent 管理
- **Sprint 2**:知识库、记忆系统
- **Sprint 3**:工具调用
- **Sprint 4**:工作流、多 Agent、Dashboard
- **Sprint 5**:桌面端

预计十周完成首个可用版本。

## 第十部分:MVP 成功后的下一步

验证用户是否每天都会打开,哪些 Agent 最常用,哪些工作真正适合交给 Agent,基于真实使用数据,再决定是否开发 Agent Forge 和 3D 功能。

## 第十一部分:一句话总结

MVP 的目标不是做出最完整、最炫的平台,而是做出第一套真正能每天帮你工作的数字员工系统。

## 架构建议

在完成整个产品规划时,我建议你不要一开始就开发 Agent Forge 的 3D 拼装系统,虽然它会是未来非常有辨识度的功能,但从产品验证角度,更合理的路径是先把 Agent Runtime 跑通,验证记忆、知识、工具和协作是否真的解决问题,再把 Agent Studio 做成培养中心,最后再推出 Agent Forge,这样可以先验证核心价值,再投入更重的 3D 交互开发,也更符合 MVP 的原则。
